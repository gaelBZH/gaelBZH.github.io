Lecture 5 : Création de Programmes Parallèles

Résumé complet en français — Calcul Haute Performance

1. Types de traitement parallèle

Le parallélisme peut être exploité à différents niveaux de granularité. On distingue quatre niveaux principaux :

Niveau instruction (ILP — Instruction Level Parallelism)

L'exécution simultanée de plusieurs instructions élémentaires au sein d'un même processeur, réalisée via le pipelining ou les architectures superscalaires. Ce niveau est géré automatiquement par le concepteur du processeur.

Niveau boucle (Loop Level Parallelism)

Distribution des itérations de boucles entre plusieurs processeurs via l'utilisation de threads. Ce niveau relève du programmeur et des compilateurs parallélisants.

Niveau tâche (Task Level Parallelism)

Découpage d'une tâche de calcul en sous-tâches affectées à des processeurs individuels. Cette granularité est décidée par le programmeur et dépasse généralement les capacités des compilateurs automatiques. Elle fait appel à des threads ou des processus.

Niveau programme (Job Level Parallelism)

Exécution simultanée de programmes indépendants. La responsabilité incombe aux concepteurs du système d'exploitation.

2. Étapes essentielles de la création d'un programme parallèle

La création d'un programme parallèle repose sur deux étapes fondamentales :

  1. Détection de la concurrence disponible : identifier quelles parties peuvent s'exécuter simultanément. Cette étape est souvent la plus créative.
  2. Détermination de la synchronisation nécessaire ou de l'échange de messages entre les processus ou threads. Cette étape requiert la connaissance du modèle de programmation.

3. Méthodologie de programmation parallèle — PCAM

Une méthodologie largement utilisée pour structurer la création de programmes parallèles est PCAM (Foster, 1985) :

LettreÉtapeDescriptionObjectif
PPartitionDécomposition de la tâche en sous-tâchesCréer un programme correct
CCommunicateDéfinition des communications nécessaires entre sous-tâchesCréer un programme correct
AAgglomerateAnalyse des variantes de décomposition pour regrouper les tâchesOptimiser le programme
MMapPrise en compte de l'implémentation finale et du mapping matérielOptimiser le programme

Les deux premières étapes (P et C) visent à produire un programme correct ; les deux suivantes (A et M) visent à l'optimiser.

Facteurs influençant la méthodologie

  • L'architecture matérielle cible.
  • La portée des tâches : parallélisation d'un code existant, d'un algorithme existant, ou développement d'un nouvel algorithme parallèle.

4. Stratégies de décomposition des tâches

a) Décomposition fonctionnelle

Division selon les fonctions (responsabilités) du programme. Applications typiques : applications complexes standard, problèmes d'optimisation complexes. Les tâches peuvent avoir des tailles très différentes.

b) Décomposition de domaine

Division selon l'espace du problème : l'ensemble de données est partitionné uniformément entre les tâches. Applications typiques : simulations physiques dans l'espace (division géométrique).

c) Décomposition de structures de données

Division selon les structures de données manipulées. Schémas de partitionnement pour une structure 2D :

  • 1D : par blocs de lignes (BLOCK), ou cyclique par lignes (CYCLIC)
  • 2D : blocs de lignes (BLOCK, *), blocs de colonnes (*, BLOCK), blocs 2D (BLOCK, BLOCK), cyclique (CYCLIC, CYCLIC)

d) Décomposition mixte (pipeline)

Combinaison fonctionnelle et données. Application typique : traitement en pipeline de séquences d'images vidéo.

5. Communication entre tâches

Problèmes "embarrassingly parallel" (parallélisme trivial)

Certains problèmes ne présentent aucune dépendance entre sous-tâches. La communication est minimale ou nulle (exemple : simulation Monte-Carlo, rendu d'images indépendantes).

Coûts de la communication

  • Tout échange introduit un délai (latence).
  • Pendant la communication, les ressources machine sont utilisées pour l'emballage et la transmission des données.
  • La synchronisation forcée provoque une charge inégale sur les threads.

Latence et bande passante

  • Latence (\(\lambda\)) : délai incompressible de tout échange (temps pour envoyer 0 données).
  • Bande passante (\(\beta\)) : quantité de données transmise par unité de temps.

Le temps de transmission d'un message de taille \(n\) octets :

\[ T_{\text{comm}}(n) = \lambda + \frac{n}{\beta} \]

Attention : envoyer beaucoup de petits messages amplifie considérablement la latence. Il vaut mieux regrouper les données en messages plus grands.

6. Synchronisation

La synchronisation implique souvent la sérialisation de fragments du programme — certaines parties ne peuvent pas s'exécuter en parallèle.

a) Barrière (Barrier)

  • S'applique typiquement à tous les threads d'un groupe.
  • Aucun thread ne peut dépasser le point de barrière tant que tous les autres ne l'ont pas atteint.

b) Verrous / Sémaphores (Locks / Semaphores)

  • Concernent des threads sélectionnés.
  • Peuvent être bloquants (le thread attend) ou non-bloquants (le thread continue si le verrou est occupé).
  • Utilisés pour protéger les sections critiques.

c) Opérations de communication synchrones

  • Concernent les threads impliqués dans une communication.
  • L'émetteur et le récepteur se synchronisent lors de l'échange (ex. : communications bloquantes MPI).

7. Équilibrage de charge (Load Balancing)

L'objectif est de minimiser le temps d'inactivité des threads. Le thread le plus lent détermine la performance globale :

\[ T_{\text{total}} = \max_{i} \, T_i \]

Stratégies d'équilibrage

  • Équilibrage du partitionnement des données : distribuer les données de façon que chaque tâche reçoive une charge comparable.
  • Équilibrage de la distribution des itérations de boucles.
  • Équilibrage sur systèmes hétérogènes : tenir compte des différentes vitesses des processeurs.
  • Pool de tâches (task pool) : les tâches sont stockées dans une file partagée ; chaque thread libre en prend une nouvelle. Adapte dynamiquement la charge en cas de tâches de durée variable.

8. Granularité des calculs

La granularité mesure le rapport entre le volume de calcul et le volume de communication :

\[ G = \frac{\text{Travail de calcul entre deux communications}}{\text{Coût de la communication}} \]

a) Parallélisme à grains fins (Fine-grained)

  • De petites quantités de calcul entre deux communications.
  • Facilite l'équilibrage de charge mais entraîne des coûts de communication élevés.
  • Risque : si la granularité est trop fine, le surcoût dépasse le temps de calcul → parallélisation contre-productive.

b) Parallélisme à gros grains (Coarse-grained)

  • De grandes quantités de calcul entre deux communications.
  • Réduit les surcoûts de communication, mais peut rendre l'équilibrage de charge plus difficile.

L'objectif est de maximiser :

\[ \text{Efficacité} = \frac{T_{\text{séquentiel}}}{p \cdot T_{\text{parallèle}}} \]

où \(p\) est le nombre de processeurs.

Exercices

Exercice 1 — Niveaux de parallélisme

Pour chacun des scénarios, identifiez le niveau de parallélisme approprié (ILP, boucle, tâche, programme) et justifiez :

  1. Un serveur web traitant simultanément des requêtes HTTP indépendantes.
  2. Le calcul de chaque pixel d'une image de rendu 3D (chacun indépendant).
  3. Un compilateur exécutant en pipeline l'analyse lexicale, syntaxique et sémantique.
  4. Un processeur superscalaire exécutant deux additions simultanément.
Indice
Pensez à qui contrôle chaque niveau : concepteur du processeur, compilateur/programmeur, ou système d'exploitation.
Exercice 2 — Méthodologie PCAM

Parallélisez le calcul \(\mathbf{y} = A\mathbf{x}\) (\(A \in \mathbb{R}^{n \times n}\)) via PCAM :

  1. Partition : Quelle stratégie de décomposition ? Comment diviser la tâche ?
  2. Communication : Quelles données doivent être échangées ?
  3. Agglomération : Comment regrouper les tâches pour réduire la communication ?
  4. Mapping : Comment affecter les tâches aux \(p\) processeurs ?
Indice
Décomposition par blocs de lignes (BLOCK, *). Chaque processeur calcule \(\frac{n}{p}\) lignes de \(\mathbf{y}\). Il faut le vecteur \(\mathbf{x}\) complet sur chaque processeur (broadcast).
Exercice 3 — Coût de communication et latence

Deux stratégies pour échanger 8 000 octets entre deux threads :

  • Stratégie A : 1 000 messages de 8 octets chacun.
  • Stratégie B : 1 message de 8 000 octets.

Avec \(\lambda = 10\,\mu s\) et \(\beta = 10^9\) octets/s, calculez \(T_{\text{comm}}\) pour chaque stratégie. Laquelle est la plus efficace ?

Indice
Stratégie A : \(1000 \times (10^{-5} + \frac{8}{10^9}) \approx 10\,\text{ms}\). Stratégie B : \(10^{-5} + \frac{8000}{10^9} \approx 0{,}018\,\text{ms}\). La stratégie B est ~556× plus rapide.
Exercice 4 — Décomposition de données

Pour une matrice \(A \in \mathbb{R}^{8 \times 8}\) et 4 processeurs, décrivez la distribution pour chaque schéma et indiquez quelles lignes/colonnes vont au processeur \(P_2\) :

  1. Distribution BLOCK, * (blocs de lignes contiguës)
  2. Distribution *, BLOCK (blocs de colonnes contiguës)
  3. Distribution CYCLIC, * (lignes cycliques)
  4. Distribution BLOCK, BLOCK (blocs 2D)
Indice
BLOCK, * : \(P_2\) reçoit les lignes 4–5. CYCLIC, * : \(P_2\) reçoit les lignes 2 et 6 (indice modulo 4 = 2).
Exercice 5 — Équilibrage de charge

Un programme sur \(p = 4\) processeurs avec \(T_0 = 10\), \(T_1 = 15\), \(T_2 = 12\), \(T_3 = 8\) secondes :

  1. Calculez \(T_{\text{total}} = \max_i T_i\).
  2. Calculez l'efficacité \(\text{Efficacité} = \frac{T_{\text{seq}}}{p \cdot T_{\text{total}}}\) avec \(T_{\text{seq}} = \sum_i T_i\).
  3. Quel thread est le goulot d'étranglement ? Quel serait le gain avec un équilibrage parfait ?
Indice
\(T_{\text{total}} = 15\,\text{s}\). \(T_{\text{seq}} = 45\,\text{s}\). Efficacité \(= \frac{45}{4 \times 15} = 75\%\). Avec équilibrage parfait : \(T_{\text{idéal}} = \frac{45}{4} = 11{,}25\,\text{s}\).
Exercice 6 — Synchronisation par barrière

Un programme parallèle utilise une barrière pour synchroniser \(p\) threads :

  1. Expliquez pourquoi la barrière peut créer un déséquilibre même si les tâches initiales sont équilibrées.
  2. Différenciez les trois types de synchronisation (barrière, verrous, comm. synchrones) avec un exemple concret pour chacun.
  3. Dans quel cas préférerait-on un verrou non-bloquant ?
Indice
Les variations de charge dues au scheduling OS, aux cache miss, etc. peuvent désynchroniser des tâches initialement équilibrées. Un verrou non-bloquant est préférable quand le thread peut effectuer un travail alternatif pendant l'attente.
Exercice 7 — Granularité et choix de décomposition

On parallélise une DFT de \(N = 1024\) points sur \(p = 8\) processeurs avec \(t_c = 1\,\text{ns}\) par calcul et \(t_{\text{comm}} = 100\,\text{ns}\) par communication.

  1. Calculez la granularité \(G\) pour \(n_{\text{loc}} = 128\) points par processeur (1 communication par tâche).
  2. Grains fins ou grossiers ? Justifiez.
  3. Si on réduit à 8 points par processeur (\(p = 128\)), comment évolue \(G\) ? Quelles conséquences ?
Indice
\(G = \frac{128 \times 1\,\text{ns}}{100\,\text{ns}} = 1{,}28\). Valeur proche de 1 → équilibre. Pour 8 points : \(G = \frac{8}{100} = 0{,}08\) → grains très fins, surcoût de communication dominant.
Exercice 8 — Pool de tâches et allocation dynamique

On traite \(N = 100\) images indépendantes (durées inconnues, entre 1 et 10 s) sur \(p = 4\) processeurs.

  1. Pourquoi une décomposition statique (25 images/processeur) peut-elle être sous-optimale ?
  2. Décrivez le fonctionnement d'un pool de tâches et expliquez pourquoi il résout ce problème.
  3. Quels sont les inconvénients potentiels d'un pool de tâches ?
  4. Ce problème est-il "embarrassingly parallel" ? Justifiez.
Indice
Si les 25 images les plus longues tombent sur un seul processeur, les autres seront en attente. Le pool de tâches résout ce problème par allocation dynamique. Inconvénient : accès concurrent à la file partagée nécessite un verrou → surcoût si les tâches sont très petites.
