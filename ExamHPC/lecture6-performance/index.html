Lecture 6a : Mesures de Performance

Résumé complet en français — Calcul Haute Performance

1. Performance de pointe (Peak Performance)

La performance de pointe théorique d'un processeur est la valeur maximale atteignable dans les conditions idéales :

\[ P_{\text{crête}} = f \times n \quad [\text{GFLOPS}] \]

où \(f\) est la fréquence d'horloge (en GHz) et \(n\) est le nombre d'opérations flottantes par cycle (dépend des unités SIMD, FMA, etc.).

Cette valeur est rarement atteignable en pratique : accès mémoire, branches, dépendances de données la limitent.

Unités de mesure

UnitéValeurContexte typique
MFLOPS\(10^6\) FLOP/sStations de travail années 1990
GFLOPS\(10^9\) FLOP/sCPU modernes
TFLOPS\(10^{12}\) FLOP/sGPU, clusters HPC
PFLOPS\(10^{15}\) FLOP/sTop 10 superordinateurs

2. Types de benchmarks

Classification par origine

  • Informels : benchmarks personnels, créés pour un besoin précis (ex. mesurer la bande passante mémoire d'un système particulier).
  • Industriels / de référence : développés par des consortiums ou organismes indépendants pour permettre des comparaisons standardisées.
  • Personnalisés : adaptés au domaine applicatif visé (ex. simulation météorologique, rendu 3D).

Classification par type

  • Algorithmiques : mesurent les performances d'algorithmes spécifiques (LINPACK → BLAS niveau 3).
  • Applicatifs : simulés des charges de travail réelles d'applications entières (SPEC CPU2006).
  • De spécification : normes utilisées pour des appels d'offres, permettant de comparer des offres matérielles et logicielles (TPC).

3. LINPACK

LINPACK résout le système linéaire \(A\mathbf{x} = \mathbf{b}\) (\(A \in \mathbb{R}^{n \times n}\)) par élimination de Gauss avec pivotage partiel :

\[ R = \frac{\frac{2}{3}n^3}{T_{\text{résolution}}} \quad [\text{GFLOPS}] \]

Le numérateur \(\frac{2}{3}n^3\) correspond au nombre d'opérations flottantes effectuées lors de la décomposition LU d'une matrice \(n \times n\).

Deux variantes principales

VarianteTaille \(n\)Utilisation
Benchmark DP\(n = 100\)Benchmark "historique" original
HPL (Highly Parallel LINPACK) / TPP\(n = 1000\) (ou très grand)Mesure des superordinateurs TOP500

LINPACK est un benchmark dense : il stresse intensément les unités flottantes et profite bien de la hiérarchie de cache via BLAS-3. Il n'est pas représentatif des applications creuses (sparse).

4. STREAM

STREAM mesure la bande passante mémoire durable (sustainable memory bandwidth). Il comprend 4 noyaux simples sur des tableaux de taille \(N\) :

NoyauOpérationBytes/iterFLOP/iter
COPY\(c[i] = a[i]\)160
SCALE\(b[i] = q \cdot a[i]\)161
SUM\(c[i] = a[i] + b[i]\)241
TRIAD\(a[i] = b[i] + q \cdot c[i]\)242

La bande passante mesurée pour chaque noyau :

\[ BW = \frac{N \times \text{bytes\_per\_iter}}{T} \]

STREAM donne une valeur pratique de la bande passante, généralement bien inférieure à la valeur théorique du fabricant.

5. SPEC CPU

SPEC (Standard Performance Evaluation Corporation) propose des benchmarks applicatifs complets :

  • SPECint : suite d'applications entières (compression, compilateurs, moteurs de recherche).
  • SPECfp : suite d'applications flottantes (CFD, météo, chimie quantique).

Calcul du score SPEC

Le score est la moyenne géométrique des ratios de performance sur \(k\) benchmarks :

\[ \text{Score SPEC} = \left(\prod_{i=1}^{k} \frac{T_{\text{réf},i}}{T_{\text{test},i}}\right)^{\frac{1}{k}} \]

où \(T_{\text{réf},i}\) est le temps de référence (système de base) et \(T_{\text{test},i}\) le temps sur le système testé. Un score > 1 indique un système plus rapide que la référence.

8. Modèle Roofline

Le modèle Roofline relie la performance atteignable à l'intensité arithmétique d'un programme :

\[ P \leq \min\!\left(P_{\text{crête}},\ I \times BW_{\text{mémoire}}\right) \]

où l'intensité arithmétique \(I\) est :

\[ I = \frac{\text{FLOPS}}{\text{bytes\_transférés}} \quad [\text{FLOP/byte}] \]

Deux régimes

  • Limité par la mémoire (Memory-bound) : \(I\) faible, la performance est plafonnée par \(I \times BW\). Le programme attend les données.
  • Limité par le calcul (Compute-bound) : \(I\) élevé, la performance est plafonnée par \(P_{\text{crête}}\). Les unités de calcul sont saturées.

Le point de croisement est appelé ridge point :\(I^* = \frac{P_{\text{crête}}}{BW}\). Au-delà de \(I^*\), le programme est compute-bound.

Utilité

Le Roofline permet d'identifier rapidement si des optimisations doivent cibler l'intensité arithmétique (transformer les accès mémoire) ou l'utilisation des unités de calcul (vectorisation, FMA).

Exercices

Exercice 1 — Performance de pointe

Un processeur fonctionne à \(f = 3{,}5\) GHz avec 8 cœurs, chacun pouvant exécuter 2 opérations SIMD de 4 flottants doubles par cycle :

  1. Calculez \(P_{\text{crête}}\) en GFLOPS.
  2. Si le processeur exécute une DGEMM atteignant 80 % de la performance de pointe, quelle est la performance effective ?
  3. Combien de temps faut-il pour factoriser LU une matrice \(n = 10\,000\) ?
Indice
\(P_{\text{crête}} = 3{,}5 \times 8 \times 2 \times 4 = 224\) GFLOPS. Ops LU : \(\frac{2}{3} \times 10^4 \approx \frac{2}{3} \times 10^{12}\) flops.
Exercice 2 — LINPACK et TOP500

Un système résout LINPACK HPL (\(n = 10\,000\)) en \(T = 2\) secondes :

  1. Calculez la performance LINPACK \(R\) en GFLOPS.
  2. Si \(P_{\text{crête}} = 400\) GFLOPS, quelle est l'efficacité LINPACK (\(\frac{R}{P_{\text{crête}}}\)) ?
  3. Pourquoi LINPACK n'est-il pas représentatif des applications scientifiques creuses ?
Indice
\(R = \frac{2/3 \times (10^4)^3}{2} = \frac{6{,}67 \times 10^{11}}{2} \approx 333\) GFLOPS. Efficacité \(\approx 83\%\). LINPACK est 100 % dense.
Exercice 3 — Benchmark STREAM

On exécute STREAM TRIAD (\(a[i] = b[i] + q \cdot c[i]\)) sur \(N = 10^8\) éléments double précision en \(T = 0{,}3\) s :

  1. Calculez la bande passante mesurée \(BW\) en Go/s.
  2. Combien de FLOPS sont effectués ? Quelle est l'intensité arithmétique \(I\) ?
  3. Avec \(P_{\text{crête}} = 100\) GFLOPS, ce noyau est-il compute-bound ou memory-bound ?
Indice
TRIAD : 24 bytes/iter. \(BW = \frac{10^8 \times 24}{0{,}3} = 8\) Go/s. \(I = \frac{2}{24} \approx 0{,}083\) FLOP/byte → très memory-bound.
Exercice 4 — Modèle Roofline

Un processeur a \(P_{\text{crête}} = 100\) GFLOPS et \(BW = 20\) Go/s :

  1. Calculez le ridge point \(I^*\).
  2. Pour un algorithme avec \(I = 2\) FLOP/byte, quelle est la performance maximale atteignable ?
  3. Pour \(I = 10\) FLOP/byte ?
  4. Quelle optimisation serait la plus utile dans chaque cas ?
Indice
\(I^* = \frac{100}{20} = 5\) FLOP/byte. Pour \(I = 2\) : \(P_{\max} = 2 \times 20 = 40\) GFLOPS (memory-bound). Pour \(I = 10\) : \(P_{\max} = 100\) GFLOPS (compute-bound).
Exercice 5 — Score SPEC

Un système est évalué sur 3 benchmarks SPEC avec les temps suivants :

Benchmark\(T_{\text{réf}}\) (s)\(T_{\text{test}}\) (s)
Benchmark A1 000250
Benchmark B500200
Benchmark C2 000400
  1. Calculez le ratio de performance pour chaque benchmark.
  2. Calculez le score SPEC (moyenne géométrique).
  3. Pourquoi utilise-t-on la moyenne géométrique plutôt qu'arithmétique pour SPEC ?
Indice
Ratios : 4, 2{,}5, 5. Score SPEC \(= (4 \times 2{,}5 \times 5)^{1/3} = 50^{1/3} \approx 3{,}68\). La moyenne géométrique est invariante par normalisation de référence.
Exercice 6 — Choix du benchmark

Pour chaque cas d'usage, identifiez le benchmark le plus approprié parmi : LINPACK, STREAM, SPEC CPU, TPC-C, HPCG, Graph500, 3DMARK.

  1. Évaluation d'un serveur de base de données OLTP.
  2. Classement d'un superordinateur dans TOP500.
  3. Mesure du goulot d'étranglement mémoire d'une application CFD creuse.
  4. Comparaison de GPUs pour jeux vidéo.
  5. Évaluation d'un système pour analyse de réseaux sociaux à grande échelle.
Indice
TPC-C, LINPACK HPL, HPCG (ou STREAM), 3DMARK, Graph500.
Exercice 7 — Intensité arithmétique et optimisation

Considérez le produit scalaire \(s = \sum_{i=0}^{N-1} a_i \cdot b_i\) avec \(N = 10^6\) doubles :

  1. Comptez le nombre de FLOPS et d'accès mémoire (en bytes).
  2. Calculez \(I\).
  3. Avec \(BW = 40\) Go/s et \(P_{\text{crête}} = 200\) GFLOPS, la performance est-elle limitée par le calcul ou la mémoire ?
  4. Proposez une modification algorithmique pour augmenter \(I\) (ex. fusion de noyaux).
Indice
FLOPS : \(2N = 2 \times 10^6\). Bytes : \(2 \times 8N = 16 \times 10^6\). \(I = \frac{2}{16} = 0{,}125\) FLOP/byte. Très memory-bound : \(P_{\max} = 0{,}125 \times 40 = 5\) GFLOPS.
Exercice 8 — Analyse comparative

Deux architectures A et B :

Système\(P_{\text{crête}}\)\(BW\)Score LINPACK
A200 GFLOPS10 Go/s160 GFLOPS
B100 GFLOPS50 Go/s80 GFLOPS
  1. Calculez le ridge point de chaque système.
  2. Lequel est préférable pour des applications memory-bound (\(I = 1\)) ?
  3. Lequel est préférable pour des applications compute-bound (\(I = 20\)) ?
  4. Pourquoi le score LINPACK seul est-il insuffisant pour choisir un système ?
Indice
Ridge A : \(200/10 = 20\) FLOP/byte. Ridge B : \(100/50 = 2\) FLOP/byte. Pour memory-bound (\(I=1\)) : B est meilleur (50 GFLOPS vs 10 GFLOPS). Pour compute-bound (\(I=20\)) : A atteint 200 GFLOPS vs 100 pour B.
