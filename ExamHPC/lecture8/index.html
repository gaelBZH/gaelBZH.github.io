Lecture 8 : Parallélisation en Mémoire Partagée

High Performance Computing — Résumé complet en français

1. Concurrence et problèmes associés

Dans un modèle à mémoire partagée, plusieurs threads s'exécutent en parallèle et accèdent aux mêmes données. Cette situation engendre des problèmes de concurrence qu'il faut impérativement maîtriser.

1.1 Condition de course (Race Condition)

Une condition de course survient lorsque le résultat d'un calcul dépend de l'ordre d'exécution non déterministe des threads. Si deux threads lisent et modifient la même variable simultanément sans protection, le résultat final est imprévisible.

Exemple : si \( p \) threads incrémentent tous une variable partagée x de façon non protégée, au lieu d'obtenir \( x + p \), on peut obtenir n'importe quelle valeur entre \( x + 1 \) et \( x + p \).

1.2 Synchronisation

La synchronisation permet de coordonner l'exécution des threads. Elle peut être réalisée :

  • Au niveau matériel — via des instructions atomiques du processeur.
  • Au niveau logiciel — via des barrières, des sections critiques et des opérations atomiques.

1.3 Exclusion mutuelle

L'exclusion mutuelle garantit qu'un seul thread à la fois peut exécuter une portion de code sensible appelée section critique.

1.4 Propriétés désirables et erreurs possibles

Une bonne synchronisation doit satisfaire :

  • Vivacité (Liveness) — le système progresse toujours, aucun thread ne reste bloqué indéfiniment.
  • Sûreté (Safety) — jamais deux threads dans la section critique simultanément.
  • Intégrité (Integrity) — les données partagées restent dans un état cohérent.

Les erreurs classiques à éviter sont :

  • Deadlock (interblocage) — deux threads ou plus s'attendent mutuellement, aucun ne peut avancer.
  • Starvation (famine) — un thread est continuellement privé d'accès à la ressource dont il a besoin.

2. OpenMP — Présentation générale

OpenMP (Open Multi-Processing) est une API standard pour la programmation parallèle en mémoire partagée. C'est l'outil de référence pour paralléliser des programmes C, C++ et Fortran sur des architectures multi-cœurs.

2.1 Caractéristiques principales

  • Portabilité — le code OpenMP peut être compilé sur la plupart des plateformes sans modification.
  • Modèle SPMD (Single Program, Multiple Data) — un seul programme est écrit, mais il est exécuté par plusieurs threads qui travaillent chacun sur une partie des données.
  • Intégré dans la plupart des compilateurs — il suffit d'activer un flag de compilation (par exemple -fopenmp avec GCC).

2.2 Composants d'OpenMP

  1. Directives de compilateur (#pragma omp ...) — instructions insérées dans le code source.
  2. Fonctions de bibliothèque — fonctions déclarées dans omp.h.
  3. Variables d'environnement — variables système qui configurent le comportement d'OpenMP (ex. OMP_NUM_THREADS).

2.3 Limites importantes

La spécification OpenMP définit l'implémentation des directives et fonctions, mais ne garantit pas la correction des programmes. C'est à lui de s'assurer que la parallélisation est correcte.

Avantage : un programme OpenMP peut toujours être compilé et exécuté de façon séquentielle (en ignorant les directives), ce qui facilite le débogage.

3. Syntaxe des directives OpenMP

3.1 Format général

Pour les langages C et C++, la syntaxe d'une directive OpenMP est :

#pragma omp <directive> [liste de clauses]

Les directives les plus importantes sont les construits de partage de travail (work-sharing constructs). Les clauses définissent le traitement des variables par les threads dans la région parallèle.

3.2 Directive parallel

La directive fondamentale est parallel, qui crée une région parallèle :

#pragma omp parallel [liste de clauses]
{
    /* région parallèle */
}

Les clauses possibles incluent :

  • if(condition) — la région n'est parallèle que si la condition est vraie.
  • num_threads(constante) — fixe le nombre de threads.
  • Attributs de partage des données : private, firstprivate, shared, reduction.
  • schedule — définit la politique d'ordonnancement des itérations.

4. Contrôle du nombre de threads

Le nombre de threads peut être spécifié de trois façons, par ordre de priorité décroissante :

MéthodeExemple
Clause num_threads dans la directive #pragma omp parallel num_threads(10)
Appel à la fonction de bibliothèque omp_set_num_threads(10);
Variable d'environnement $ set OMP_NUM_THREADS=10

Exemple minimal

#include <stdio.h>
#include <omp.h>

int main()
{
    #pragma omp parallel
    {
        printf("I am thread %d\n", omp_get_thread_num());
    }
}

Ce programme crée autant de threads que disponibles. Chaque thread imprime son identifiant, qui va de \(0\) à \(p-1\). L'ordre d'affichage est non déterministe.

6. Construits de partage de travail (Work-sharing constructs)

Les construits de partage de travail permettent de distribuer le travail entre les threads d'une équipe. Règles fondamentales :

  • Tous les threads d'une équipe doivent rencontrer les mêmes directives de partage de travail.
  • Il n'y a pas de barrière implicite à l'entrée d'un construit de partage de travail.

6.1 Construit for

Le construit for distribue les itérations d'une boucle entre les threads :

#pragma omp for [liste de clauses]
for (...) { ... }   // boucle en forme canonique

Les clauses disponibles pour for incluent :

  • Partage de données : private, firstprivate, lastprivate
  • Ordonnancement : schedule(type, taille_de_bloc) avec les types :
    • static — les itérations sont divisées en blocs de taille fixe et assignées de façon cyclique.
    • dynamic — les blocs sont assignés dynamiquement à la demande.
    • guided — blocs de taille décroissante assignés dynamiquement.
    • runtime — la politique est déterminée à l'exécution via OMP_SCHEDULE.
  • reduction, nowait

La combinaison fréquente parallel for fusionne les deux directives en une seule :

#pragma omp parallel for [liste de clauses]
for (...) { ... }

6.2 Autres directives importantes

DirectiveSyntaxeRôle
master #pragma omp master { ... } Seul le thread maître (id=0) exécute le bloc
critical #pragma omp critical { ... } Section critique : un seul thread à la fois peut exécuter ce bloc
barrier #pragma omp barrier Point de synchronisation : tous les threads attendent ici

7. Clauses de partage des variables

Le traitement des variables dans une région parallèle est contrôlé par des clauses de data sharing.

ClauseSignification
shared(var) La variable est partagée entre tous les threads — tous voient et modifient la même case mémoire.
private(var) Chaque thread dispose d'une copie locale non initialisée de la variable.
firstprivate(var) Chaque thread dispose d'une copie locale initialisée avec la valeur de la variable avant l'entrée dans la région parallèle.
lastprivate(var) Variable locale à chaque thread, mais après la région parallèle, la valeur finale correspond à celle de la dernière itération séquentielle.
reduction(op:var) Chaque thread calcule une valeur partielle avec l'opérateur op, et les résultats partiels sont combinés à la fin de la région.

Exemple illustratif (private / firstprivate / lastprivate)

Avec #define N 3 et les valeurs initiales a=1, b=2, c=3, d=4 :

#pragma omp parallel for private(b) firstprivate(c) lastprivate(d)
for(i = 0; i < N; i++)
{
    b += 2;
    c += 2;
    d = omp_get_thread_num();
}
  • b est private : chaque thread a une copie non initialisée (valeur indéfinie).
  • c est firstprivate : chaque thread commence avec c = 3. Hors de la région, c conserve sa valeur originale 3.
  • d est lastprivate : après la région, d prend la valeur assignée lors de la dernière itération séquentielle (d = N-1 = 2).

8. La clause reduction

La clause reduction est un mécanisme très puissant pour les opérations d'accumulation parallèle. Elle crée une copie locale pour chaque thread, puis combine toutes les copies à la fin via l'opérateur spécifié.

Syntaxe : reduction(opérateur : variable)

Opérateurs supportés : +, -, *, &, |, ^, &&, ||

Exemple : somme parallèle

On souhaite calculer \(\displaystyle S = \sum_{i=0}^{100} i\).

#include <omp.h>
#define NUM 100
#define THR 4

int main(int argc, char *argv[])
{
    int i, sum = 0;
    #pragma omp parallel for reduction(+:sum) num_threads(THR)
    for(i = 0; i <= NUM; i++)
    {
        sum += i;
    }
}

Chacun des 4 threads calcule une somme partielle, puis OpenMP additionne automatiquement les 4 sommes partielles. On attend \( S = \frac{100 \times 101}{2} = 5050 \).

Sans la clause reduction, l'incrémentation sum += i serait une condition de course.

Exercices

Exercice 1 — Identification des problèmes de concurrence

Considérez un programme où \(p = 4\) threads exécutent simultanément l'instruction x = x + 1 sur une variable partagée x initialisée à 0. Expliquez pourquoi le résultat final peut être différent de 4. Quelles sont les valeurs possibles de x ? Proposez une correction en OpenMP.

Indice
L'instruction x = x + 1 se décompose en trois étapes (lecture, incrémentation, écriture). Si deux threads lisent la même valeur avant que l'un n'écrive, un incrément est perdu. Utilisez #pragma omp atomic ou #pragma omp critical.
Exercice 2 — Premier programme OpenMP

Écrivez un programme C utilisant OpenMP qui crée 8 threads. Chaque thread doit afficher son identifiant et le nombre total de threads. Vérifiez que l'ordre d'affichage n'est pas garanti. Quelle fonction OpenMP permet d'obtenir l'identifiant du thread maître ?

Indice
Utilisez omp_get_thread_num() et omp_get_num_threads() dans une région #pragma omp parallel num_threads(8). Le thread maître a toujours l'identifiant 0.
Exercice 3 — Différence entre private, firstprivate et lastprivate

Soit le code suivant avec N = 4 et les valeurs initiales a = 10, b = 20, c = 30 :

#pragma omp parallel for private(a) firstprivate(b) lastprivate(c)
for(int i = 0; i < N; i++) {
    a = i * 2;
    b = b + i;
    c = i;
}

(a) Quelle est la valeur de a après la région parallèle ?
(b) Quelle est la valeur de b dans chaque thread, juste après l'entrée dans la région ?
(c) Quelle est la valeur de c après la région parallèle ?

Indice
(a) a est private donc non défini hors de la région. (b) b est firstprivate donc initialisé à 20 dans chaque thread. (c) c est lastprivate : sa valeur finale est celle de la dernière itération séquentielle, soit i = N-1 = 3, donc c = 3.
Exercice 4 — Calcul parallèle d'une somme avec reduction

Écrivez un programme OpenMP en C pour calculer le produit scalaire de deux vecteurs \(\mathbf{u}\) et \(\mathbf{v}\) de taille \(n = 1000\) :

\[ \langle \mathbf{u}, \mathbf{v} \rangle = \sum_{i=0}^{n-1} u_i \cdot v_i \]

Utilisez la clause reduction avec 4 threads. Comparez avec une version utilisant #pragma omp critical : laquelle est plus efficace et pourquoi ?

Indice
La clause reduction(+:dot) évite les accès concurrents sans verrouillage explicite. Avec critical, les threads se mettent en file d'attente à chaque itération. La reduction est bien plus efficace car chaque thread accumule localement.
Exercice 5 — Politique d'ordonnancement (scheduling)

Expliquez la différence entre les politiques static, dynamic et guided pour la clause schedule d'OpenMP. En particulier, si une boucle de \(n = 120\) itérations est distribuée entre \(p = 4\) threads avec schedule(static, 10), décrivez précisément quelles itérations sont assignées à chaque thread.

Indice
Avec static et un bloc de taille 10 : thread 0 reçoit les itérations 0–9 et 40–49 et 80–89, thread 1 reçoit 10–19 et 50–59 et 90–99, etc. (distribution cyclique par blocs). dynamic convient aux boucles à charge inégale ; guided démarre avec de grands blocs qui diminuent pour équilibrer la charge en fin de boucle.
Exercice 6 — Deadlock et section critique

Deux threads A et B exécutent le code suivant dans une région parallèle :

// Thread A                    // Thread B
#pragma omp critical(lock1)   #pragma omp critical(lock2)
{                             {
  #pragma omp critical(lock2)   #pragma omp critical(lock1)
  { /* travail */ }             { /* travail */ }
}                             }

(a) Identifiez le problème. (b) Quelle propriété de la synchronisation est violée ? (c) Proposez une solution.

Indice
C'est un deadlock : A détient lock1 et attend lock2, pendant que B détient lock2 et attend lock1. La propriété de vivacité (liveness) est violée. Solution : imposer un ordre global d'acquisition des verrous (toujours acquérir lock1 avant lock2).
Exercice 7 — Parallélisation d'une multiplication matrice-vecteur

Soit une matrice \(A \in \mathbb{R}^{m \times n}\) et un vecteur \(\mathbf{x} \in \mathbb{R}^n\). On souhaite calculer \(\mathbf{y} = A\mathbf{x}\) où \(y_i = \sum_{j=0}^{n-1} A_{ij} \cdot x_j\).

Écrivez le code C+OpenMP en parallélisant la boucle externe sur \(i\) avec \(p\) threads. Quelles variables doivent être shared et quelles variables doivent être private ? Pourquoi n'a-t-on pas besoin de reduction ici ?

Indice
A, x, y sont shared. L'indice interne j est private. Pas besoin de reduction car chaque thread écrit dans une case y[i] différente — il n'y a pas de conflit d'écriture.
Exercice 8 — Analyse et correction d'un programme OpenMP

Analysez le programme suivant et identifiez tous les problèmes :

#include <omp.h>
int global_sum = 0;

int main() {
    int i;
    #pragma omp parallel for shared(global_sum)
    for(i = 0; i < 1000; i++) {
        global_sum += i;   // (1)
    }

    int x = 5;
    #pragma omp parallel private(x)
    {
        x = omp_get_thread_num() * 10;
    }
    printf("%d\n", x);   // (2)
}

(a) L'instruction (1) est-elle correcte ? Proposez une correction.
(b) Que va afficher l'instruction (2) et pourquoi est-ce un comportement indéfini ? Comment corriger cela ?

Indice
(a) Non : global_sum += i est une condition de course. Correction : utiliser reduction(+:global_sum) ou entourer avec #pragma omp atomic. (b) x est private, donc hors de la région parallèle, x retrouve sa valeur initiale de 5 (la copie locale est détruite). Pour récupérer la valeur du thread maître, utiliser lastprivate(x) ou un bloc #pragma omp master.
