Lecture 9 : Programmation GPU

Résumé complet en français — High Performance Computing

1. Histoire du GPGPU

Le terme GPGPU (General Purpose computing on Graphics Processing Units) désigne l'utilisation des processeurs graphiques pour des calculs scientifiques généraux, au-delà du rendu 3D.

Jalons historiques

  • 1999 — Nvidia GeForce 256 : premier GPU commercialisé. L'unité est entièrement câblée et dédiée au rendu graphique.
  • ATI Radeon 256 : introduction d'un premier type de shader programmable.
  • Nvidia GeForce 3 : généralisation des shaders programmables ; les développeurs peuvent écrire leurs propres programmes s'exécutant sur le GPU.
  • G80 (GeForce 8800, 2006) : architecture unifiée avec des multiprocesseurs de flux (Streaming Multiprocessors, SM). C'est la base de CUDA. Le G80 introduit un modèle de programmation généraliste sur GPU.

2. Architecture GPU moderne

Un GPU moderne est organisé en une hiérarchie de niveaux matériels. L'architecture G80 d'Nvidia illustre bien ce principe :

  • Le CPU hôte communique avec le GPU via un bridge et une interface hôte.
  • Un Input Assembler reçoit les données ; un module de rasterisation et clip gère la géométrie.
  • Le travail est distribué à des TPC (Texture Processing Clusters), chacun contenant plusieurs SM (Streaming Multiprocessors).
  • Chaque SM contient plusieurs Stream Processors (SP), une mémoire partagée, un cache d'instructions et un décodeur.
  • Un réseau d'interconnexion relie les SM à la mémoire globale (DRAM).

AMD RDNA

L'architecture AMD RDNA utilise des Compute Units contenant des unités vectorielles SIMD duales, des unités scalaires, des accélérateurs IA et des accélérateurs de ray-tracing. Les puces RDNA récentes adoptent une topologie multi-chiplets reliés par Infinity Cache.

Intel Xe

Intel Xe organise ses GPU en Render Slices, chacun composé de plusieurs Xe-cores. Un Xe-core intègre :

  • 16 Vector Engines (256 bits chacun) pour le calcul SIMD flottant
  • 16 Matrix Engines XMX (1024 bits) pour l'inférence IA et les opérations tensorielles
  • Un cache L1/SLM (Shared Local Memory)

4. Modèles de programmation pour accélérateurs GPU

Plusieurs modèles permettent d'exploiter la puissance des GPU :

  • CUDA (Compute Unified Device Architecture) — exclusif aux GPU Nvidia. Modèle le plus répandu et le plus performant pour les GPU Nvidia ; offre un contrôle fin de la mémoire et du parallélisme.
  • Offload — limité aux accélérateurs Intel Xeon Phi. Permet de déporter des régions de code sur le coprocesseur via des directives.
  • OpenACC — modèle universel et portable, basé sur des directives similaires à OpenMP. Facile d'utilisation. Convient à une large gamme de matériel.
  • OpenCL — standard ouvert, portable sur tous les GPU (Nvidia, AMD, Intel) et même sur CPU. Idéal pour la portabilité.

5. Modèle de programmation GPU

Hiérarchie : threads → warps → blocs → grille

  • Thread : unité d'exécution élémentaire. Chaque thread exécute le même code kernel mais sur des données différentes.
  • Warp : groupe de 32 threads exécutés simultanément en SIMD sur un même SM (chez Nvidia). Tous les threads d'un warp exécutent la même instruction au même instant (SIMT — Single Instruction Multiple Threads).
  • Bloc : ensemble de warps assigné à un même SM. Les threads d'un bloc partagent la mémoire partagée et peuvent se synchroniser.
  • Grille (Grid) : ensemble de blocs lancés par un même appel de kernel. La grille peut être 1D, 2D ou 3D.

Différences entre threads GPU et threads CPU

  • Les threads GPU sont extrêmement légers : leur création prend un temps négligeable.
  • Un CPU multi-cœur fonctionne correctement avec quelques dizaines de threads ; un GPU a besoin de milliers de threads pour atteindre son plein potentiel (latence mémoire masquée par le changement de warp).
  • Le GPU est un coprocesseur : il n'exécute pas de code hôte, il reçoit ses instructions du CPU.

6. CUDA — Compute Unified Device Architecture

Définition et principes

CUDA est le modèle de programmation parallèle propriétaire de Nvidia. Il étend le langage C/C++ avec des constructions permettant d'exécuter des kernels (noyaux de calcul) sur le GPU.

  • Le GPU est vu comme un coprocesseur massivement parallèle dédié aux calculs intensifs.
  • Le CPU (hôte) orchestre l'exécution : il transfère les données, lance les kernels et récupère les résultats.
  • Le GPU dispose de sa propre mémoire DRAM (device memory), distincte de la mémoire principale du CPU.
  • La gestion de la mémoire GPU est explicite : le programmeur contrôle les allocations et les transferts.

Flux d'exécution CUDA (Processing Flow)

  1. Copier les données de traitement depuis la mémoire principale vers la mémoire GPU.
  2. Le CPU instruit le GPU de lancer le kernel de calcul.
  3. Le GPU exécute le kernel en parallèle sur ses milliers de cœurs.
  4. Copier les résultats depuis la mémoire GPU vers la mémoire principale.

Modèle de code : code hôte et kernels

// Code série (host)
KernelA<<< nBlk, nTid >>>(args);  // Lancement kernel A (device)
// Code série (host)
KernelB<<< nBlk, nTid >>>(args);  // Lancement kernel B (device)

Les kernels sont asynchrones par défaut : le CPU continue son exécution pendant que le GPU traite. Une synchronisation explicite (cudaDeviceSynchronize()) est nécessaire pour bloquer le CPU jusqu'à la fin du kernel.

7. Mots-clés CUDA — Qualificateurs de fonctions

CUDA ajoute trois qualificateurs de fonctions au C/C++ :

QualificateurExécuté surAppelable depuisUsage
__device__ GPU (device) GPU (device) uniquement Fonctions auxiliaires appelées par un kernel
__global__ GPU (device) CPU (host) uniquement Kernels lancés depuis le CPU
__host__ CPU (host) CPU (host) uniquement Fonctions CPU classiques (comportement par défaut)

Exemple de déclaration d'un kernel :

__global__ void KernelFunc(float* A, float* B, float* C, int N) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < N) C[i] = A[i] + B[i];
}

8. Identifiants de threads et hiérarchie de la grille

Chaque thread possède un identifiant unique dans sa grille, accessible via des variables intégrées :

  • threadIdx.x / .y / .z — position du thread dans son bloc
  • blockIdx.x / .y — position du bloc dans la grille
  • blockDim.x / .y / .z — dimensions du bloc en nombre de threads
  • gridDim.x / .y — dimensions de la grille en nombre de blocs

Pour un tableau 1D de taille \(N\), l'index global d'un thread est :

\[ i = \text{blockIdx.x} \times \text{blockDim.x} + \text{threadIdx.x} \]

Pour un tableau 2D de dimensions \(N_x \times N_y\), les indices globaux sont :

\[ i = \text{blockIdx.x} \times \text{blockDim.x} + \text{threadIdx.x} \] \[ j = \text{blockIdx.y} \times \text{blockDim.y} + \text{threadIdx.y} \]

Un kernel produit exactement une grille. La grille contient \(N_{\text{blocs}}\) blocs, chaque bloc contient jusqu'à \(N_{\text{threads}}\) threads.

9. Modèle mémoire CUDA

CUDA définit une hiérarchie de mémoires, chacune avec une portée et des performances différentes :

Type de mémoirePortéeDurée de vieVitesse
Registres Un thread Durée du thread Très rapide (sur puce)
Mémoire partagée Tous les threads d'un bloc Durée du bloc Rapide (on-chip, ~100x plus rapide que globale)
Mémoire globale Tous les threads + hôte Durée de l'application Lente (DRAM, latence ~400–800 cycles)
Mémoire constante Tous les threads (lecture seule) Durée de l'application Rapide si accès uniforme (cache dédié)
Mémoire de texture Tous les threads (lecture seule) Durée de l'application Cache optimisé pour la localité 2D

10. Gestion de la mémoire GPU en CUDA

Allocation et libération : cudaMalloc / cudaFree

  • cudaMalloc(void** ptr, size_t size) — alloue size octets dans la mémoire globale du device.
  • cudaFree(void* ptr) — libère la mémoire précédemment allouée sur le device.

Exemple d'allocation d'une matrice \(64 \times 64\) de flottants simples précision :

const int TILE_WIDTH = 64;
float* Md;
int size = TILE_WIDTH * TILE_WIDTH * sizeof(float);  // = 16384 octets

cudaMalloc((void**)&Md, size);
// ... utilisation de Md dans les kernels ...
cudaFree(Md);

La taille totale allouée est \(\text{TILE\_WIDTH}^2 \times \text{sizeof(float)} = 64^2 \times 4 = 16\,384\) octets \(= 16\) Ko.

Transfert de données : cudaMemcpy

cudaMemcpy(void* dst, void* src, size_t count, cudaMemcpyKind kind) — copie count octets. Les quatre directions possibles :

  • cudaMemcpyHostToHost — CPU vers CPU
  • cudaMemcpyHostToDevice — CPU vers GPU
  • cudaMemcpyDeviceToHost — GPU vers CPU
  • cudaMemcpyDeviceToDevice — GPU vers GPU
// Hôte vers Device
cudaMemcpy(Md, M, size, cudaMemcpyHostToDevice);

// Device vers Hôte
cudaMemcpy(M, Md, size, cudaMemcpyDeviceToHost);

Mémoire unifiée : cudaMallocManaged

La Unified Memory permet au CPU et au GPU d'utiliser les mêmes pointeurs. Le système de migration de pages gère automatiquement les transferts :

float *M, *N, *P;
cudaMallocManaged(&M, bytes);
cudaMallocManaged(&N, bytes);
cudaMallocManaged(&P, bytes);

MatrixMulKernel<<<dimGrid, dimBlock>>>(M, N, P, W);
cudaDeviceSynchronize();

// P est directement lisible sur CPU (sans D2H explicite)
cudaFree(M); cudaFree(N); cudaFree(P);

11. Kernels et configuration d'exécution

Syntaxe de lancement d'un kernel

Un kernel est lancé avec la syntaxe triple chevrons <<< ... >>> :

__global__ void KernelFunc(...);

dim3 DimGrid(100, 50);          // 100 * 50 = 5000 blocs de threads
dim3 DimBlock(4, 8, 8);         // 4 * 8 * 8 = 256 threads par bloc
size_t SharedMemBytes = 64;     // 64 octets de mémoire partagée dynamique

KernelFunc<<< DimGrid, DimBlock, SharedMemBytes >>>(...);

Le nombre total de threads lancés :

\[ N_{\text{total}} = N_{\text{blocs}} \times N_{\text{threads/bloc}} = (100 \times 50) \times (4 \times 8 \times 8) = 5000 \times 256 = 1\,280\,000 \text{ threads} \]

Le type dim3

dim3 est un type vecteur d'entiers non signés à 3 composantes \((x, y, z)\). Les composantes non spécifiées valent 1 par défaut.

Asynchronisme des kernels

Tout lancement de kernel est asynchrone : le CPU reprend immédiatement la main après l'appel. Pour attendre la fin d'exécution sur le GPU :

cudaDeviceSynchronize();   // bloque le CPU jusqu'à ce que le GPU ait terminé

12. Compilation CUDA avec NVCC

Tout fichier source contenant des extensions CUDA (.cu) doit être compilé avec NVCC (Nvidia CUDA Compiler Driver).

Fonctionnement de NVCC

  • NVCC est un driver de compilation : il orchestre plusieurs outils.
  • Le code GPU est compilé vers du PTX (assembleur virtuel Nvidia), puis vers du code binaire pour l'architecture cible.
  • Le code CPU est compilé par le compilateur hôte standard (g++, cl...).

Bibliothèques requises

  • cudart — bibliothèque d'exécution CUDA : fonctions comme cudaMalloc, cudaMemcpy, etc.
  • cuda — bibliothèque cœur CUDA : interface de bas niveau avec le driver GPU.
nvcc -o mon_programme mon_programme.cu -lcudart
nvcc -arch=sm_90 -o mon_programme mon_programme.cu   # pour Hopper

Exercice 2 — Index global d'un thread

Considérez un kernel 1D traitant un tableau de taille \(N = 1\,000\,000\). La configuration de lancement est : DimBlock(256).

  1. Écrivez la formule de l'index global \(i\) d'un thread.
  2. Quel est le nombre minimal de blocs nécessaires pour couvrir \(N = 1\,000\,000\) éléments ?

Exercice 5 — Mémoire unifiée vs explicite

Indice

Exercice 8 — Modèles de programmation : choix de technologie

Pour chacun des scénarios suivants, choisissez le modèle de programmation GPU le plus adapté (CUDA, OpenACC, OpenCL) et justifiez :

  1. Un laboratoire utilisant exclusivement des GPU Nvidia H100 pour simuler des écoulements fluides (CFD). La performance maximale est critique.
  2. Un chercheur qui dispose d'un cluster hétérogène avec des GPU AMD et Nvidia et veut un seul code source portable.
  3. Un ingénieur qui veut paralléliser rapidement un code Fortran existant avec un minimum de modifications, sur GPU Nvidia.
Indice
1. CUDA : performances optimales sur Nvidia, contrôle fin de la mémoire et des kernels.
